# 1、右值引用

1. 左右值**引用**本质上是引用对象的**别名**，也就是原变量的另一个名字，**不占独立内存**。

2. **右值引用**的**本质**是“识别并利用即将销毁的临时对象”——要么移动他的资源，要么转发，**目的**是提升程序的运行效率。

3. 非**const** 的左值引用(T&)只能绑定到左值对象，**const** T& s可以绑定所有值。

4. 右值 = 将亡值 + 纯右值。

5. **将亡值**本质是一个能够当作右值使用的**左值**。

6. **move**本质是把左值强制转为右值 ，本身不做任何操作。

7. 左值：有固定内存地址，可持久存在。

8. ```
   自己有内存，直接往里填，不用造临时；
   只是当别名，没地放数值，必须造临时。
   ```

==应用场景1==：优化拷贝构造性能开销大的问题

拷贝构造：分配新内存 → 拷贝数据 → 销毁临时对象（释放内存）。

通过右值引用：通过右值引用直接接管它的资源，跳过 “分配 + 拷贝”， 实现零拷贝开销。

```c++
#include <iostream>
#include <cstring>
// 必须添加：禁用 VS 的 strcpy_s 安全警告
#define _CRT_SECURE_NO_WARNINGS
using namespace std;

class String {
public:
    char* getStr() {
        return _str;
    }

    // 普通构造函数：初始化新对象
    String(const char* str = "") : _len(strlen(str)) {
        _str = new char[_len + 1];      //分配内存
        strcpy_s(_str, _len + 1, str);  //拷贝全部数据
        cout << "普通构造函数" << endl;
    }

    // 拷贝构造函数：深拷贝已有对象
    String(const String& s) : _len(s._len) {
        _str = new char[_len + 1];      //分配内存
        strcpy_s(_str, _len + 1, s._str);   ////拷贝全部数据
        cout << "拷贝构造" << endl;
    }

    // 移动构造函数：接管右值资源，_str(s._str)：让_str指向s._str的内存
    String(String&& s) noexcept : _str(s._str), _len(s._len) {
        //想象一下，有两个String对象的_str同时指向同一片内存，有一个对象析构的同时把这片内存
        //也释放掉了，那另一个对象地_str不就出错了吗，这也就是为什么置空
        s._str = nullptr; // 源对象置空，避免重复释放
        cout << "移动构造" << endl;
    }

    // 析构函数：释放动态内存
    ~String() {
        //如果为不为空才释放
        if (_str) {
            delete[] _str;
            _str = nullptr; // 置空，避免野指针
        }
        cout << "析构函数" << endl;
    }

private:
    char* _str;  // 指向动态分配的字符数组
    size_t _len; // 字符串长度（不含 '\0'）
};

// 返回临时 String 对象的函数
String getTempString(const char* str) {
    String tmp(str);  // 调用普通构造函数
    //return tmp 时，编译器会自动创建一个临时对象 ret_tmp（函数返回值的载体），并根据 tmp 的
    //类型和属性（左值 / 右值），选择用「拷贝构造」或「移动构造」初始化这个临时对象，最终把这
    //个临时对象作为函数的返回值传递给调用处。
    return tmp;     // 返回临时对象（右值）
}

//-fno-elide-constructors：禁用编译器的返回值优化
//C++17 后可能看不到多次移动构造，所以需要加上-std=c++14
//在终端 g++ -std=c++14 -fno-elide-constructors xxxx.cpp -o xxxx.exe
//然后.\xxxx.exe可以看到编译器优化前运行结果
//普通构造函数  // getTempString 中创建 tmp
//移动构造      // tmp → 函数返回的临时对象 ret_tmp（第一次移动）
//析构函数      // tmp 析构（已置空）
//移动构造      // ret_tmp → s2（第二次移动）
//析构函数      // ret_tmp 析构（已置空）
//World         // 打印 s2 的字符串
//析构函数      // s2 析构

//通过运行结果可以看到程序执行了三次析构函数，但因为源对象置空的原因，
//执行析构函数，实际上只delete[]两次，有一片内存直接被使用，避免了
//分配内存和拷贝数据

int main() {
    String s2(getTempString("World")); // 用ret_tmp临时对象初始化 s2
    //String s1 = s2;     //拷贝构造
    cout << s2.getStr() << endl;
    return 0;
}
```





# 2、智能指针

**智能指针：**就是用类封装原始指针，利用RAII机制，在对象析构时自动释放内存，避免内存泄漏

**1、unique_ptr（独占式智能指针）**

- **核心特点：**独占所指向的内存，不允许拷贝（只能移动），同一时间内只有一个unique_ptr能指向同一块内存

  ```C++
  void test1() {
      //独占所指向的内存，不允许拷贝(只能移动)，同一时间内只能有一个unique_ptr指向同一块内存
      std::unique_ptr<int> ptr1(new int(10));
      //make_unique(C++ 14以上)，更安全
      std::unique_ptr<int> ptr2 = std::make_unique<int>(20);
  
      std::cout << *ptr1 << std::endl;
      std::cout << *ptr2 << std::endl;
  
      //移动语义（唯一的“转移所有权”方式）
      std::unique_ptr<int> ptr3(std::move(ptr1)); //ptr1失去所有权，变为空
      if (ptr1 == nullptr) {
          std::cout << "ptr1 已为空" << std::endl;
      }
  
      std::cout << *ptr3 << std::endl;
  
      //无需手动释放内存，离开作用域会自动释放
  }
  ```

**2、shared_ptr（共享式智能指针）**

- **核心特点：**允许多个 `shared_ptr` 指向同一块内存，内部通过**引用计数**管理 —— 每多一个指针指向该内存，计数 + 1；每少一个，计数 - 1；当计数为 0 时，自动释放内存。

  ```C++
  void test2() {
      //允许多个share_ptr指向同一块内存，
      // 内部通过引用计数管理——每多一个指针指向该内存计数+1，反之-1，为0则自动释放该内存
      std::shared_ptr<int> ptr1 = std::make_shared<int>(30);
      std::cout << "引用计数" << ptr1.use_count() << std::endl;
  
      //拷贝，计数+1
      std::shared_ptr<int> ptr2 = ptr1;
      std::cout << *ptr2 << std::endl;
      std::cout << "引用计数" << ptr2.use_count() << std::endl;
      
      //两个指针指向同一块内存
      *ptr2 = 40;
      std::cout << *ptr1 << std::endl;
  
      //手动释放ptr2
      ptr2.reset();
      std::cout << "引用计数" << ptr1.use_count() << std::endl;
  
      //离开作用域ptr1自动释放
  }
  ```

**3、weak_ptr（弱引用智能指针）**

- **核心特点：**配合 `shared_ptr` 使用，是 “弱引用”—— 指向同一块内存，但**不增加引用计数**，不会阻止内存被释放。可以解决 `shared_ptr` 的**循环引用**问题

  ```C++
  class Student;
  
  class Teacher {
  public:
      std::shared_ptr<Student> stu_ptr;
  
      Teacher() {
          std::cout << "Teacher 构造函数" << std::endl;
      }
  
      ~Teacher() {
          std::cout << "Teacher 析构函数" << std::endl;
      }
  
  };
  
  class Student {
  public:
      std::shared_ptr<Teacher> tea_ptr;
      //std::weak_ptr<Teacher> tea_ptr;
  
      Student() {
          std::cout << "Student 构造函数" << std::endl;
      }
  
      ~Student() {
          std::cout << "Student 析构函数" << std::endl;
      }
  };
  
  //shared_ptr循环引用问题
  void test3() {
      //注：堆上的对象需显示释放，栈上的对象随作用域结束自动销毁
      //堆上创建Teacher对象，栈上创建shared_ptr对象tea
      std::shared_ptr<Teacher> tea = std::make_shared<Teacher>();
      //堆上创建Student对象，栈上创建shared_ptr对象stu
      std::shared_ptr<Student> stu = std::make_shared<Student>();
  
      std::cout << tea.use_count() << std::endl;
      std::cout << stu.use_count() << std::endl;
  
      //互相持有
      tea->stu_ptr = stu;
      stu->tea_ptr = tea;
  
      // 输出：2（main中的tea + stu中的tea_ptr）
      std::cout << tea.use_count() << std::endl;  
      // 输出：2（main中的stu + tea中的stu_ptr）
      std::cout << stu.use_count() << std::endl;
      
      //但作用域结束时，tea和stu都自动销毁，计数-1，但由于tea_ptr和stu_ptr互相持有，导致计数不能为0，无法释放内存
      //解决思路：把循环引用的其中一方（或双方）的 shared_ptr 改成 weak_ptr，打破 “闭环”，让引用计数能正常降到 0。
  }
  ```

  



# 3、操作系统

### 一、内存管理

#### 1、虚拟地址：

程序里用的地址，不是真实物理内存地址每个进程都有独立的虚拟地址空间

- 不让不同进程直接碰真实物理内存，从根本上避免多个进程互相干扰
      隔离进程，防止相互破坏
- 让进程觉得内存是连续、干净的
    	 物理内存实际是碎片的，但虚拟地址可以给进程展示连续的地址空间
- 支持内存扩容

#### 2、**物理地址：**

- 真实硬件内存的地址
- 只有操作系统和硬件 MMU 能看到
- 多个进程的虚拟地址会映射到不同物理页

#### 3、**内存管理单元（MMU）：**

- 硬件模块，集成在CPU里
- 负责虚拟地址 → 物理地址的转换
- 全程硬件加速，速度极快
- 流程：CPU给出虚拟地址 → MMU查页表 → 得到物理地址，访问内存

#### 4、**页表：**

- 操作系统维护的映射表
- 记录虚拟页号 → 物理页号
- 每个进程都有自己独立的页表

#### 5、**虚拟内存扩容原理：**

当物理内存不足时，操作系统把暂时不用的内存页写到磁盘的**Swap**分区，腾出物理内存给新页使用，下次访问时，再从**Swap**读回内存

#### 6、**内存分段：**

按程序的逻辑功能划分虚拟地址空间，每个逻辑部分称为段

- 代码段：存放程序指令，只读+执行

- 数据段：存放全局变量，可读可写

- 栈段：存放局部变量/函数调用

- 堆段：动态分配内存，可读可写

  每个段是大小不固定的连续区域，物理内存中也是按段分配连续空间

  **内存分段管理**：需要多少内存就分配多少，所有不会出现内部内存碎片，但会出现外部内存碎片，如：

![image-20260318225551589](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260318225551589.png)

#### 7、**内存分页：**

把整个虚拟和物理内存空间切成一段段固定尺寸的大小



![image-20260318230439377](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260318230439377.png)

分页管理：页与页之间是紧密排列的，所以不会有外部碎片，但分配内存的最小单位是一页，即使程序不足一页大小，我们最少只能分配一个页，所以页内会出现内存浪费

#### 8、**段页式内存管理：**

- 先按程序逻辑分段，解决逻辑隔离+权限控制
- 再把每个段分页，解决外部碎片+离散分配
- 虚拟地址 = 段号 + 段内页号 + 页内偏移
  - 段号：标识属于哪个逻辑段
  - 段内页号：该段内的虚拟页号
  - 页内偏移：页内的具体地址

![image-20260318230933772](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260318230933772.png)

#### 9、**映射表：**

- 先访问段表，得到页表起始地址
- 再访问页表，得到物理页号
- 最后将物理页号与页内位移组合，得到物理地址

![image-20260318231721724](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260318231721724.png)

#### 10、**TLB**

TLB是虚拟地址 → 物理地址的高速缓存。

在CPU芯片里，封装内存管理单元（MMU）芯片，它用来完成地址转换和TLB的访问和交互。

有了TLB之后，那么CPU在寻址时，会先查TLB，如果没没找到，才会继续查常规的页表。

TLB 的命中率其实是很高的，因为程序最常访问的页就那么几个。

### 二、**进程和线程**

#### 1、**进程是资源分配的基本单位**

一个进程被创建时，操作系统会**一次性给它分配一整套独立资源**，这些资源**归这个进程独有**，其它进程不能随便使用。

进程拥有的资源：

- 独立的**虚拟地址空间**
- 打开的**文件描述符**
- 页表、内存映射等

简单理解：

**进程 = 一个独立的 “程序容器”**

操作系统只认容器，给容器分配资源，不关心容器里跑几个执行流。



#### 2、**线程是CPU调度的单位**

CPU真正执行的不是进程，而是**线程**。

一个进程内存可以有多个线程，它们共享进程的大部分资源，但：

- 有独立的**PC（程序计数器）**
- 有独立的**栈**
- 有独立的**寄存器上下文**

这些东西决定了：

**CPU下一步执行哪条指令、执行到哪了。**

操作系统调度器看的就是线程，把CPU时间片分给线程



#### 3、线程切换比进程切换快：

**同一进程内的线程共享虚拟地址空间与页表，线程切换只需保存和恢复寄存器、栈等少量上下文，无需更换页表和刷新 TLB，因此开销远小于进程切换。**

两个不同进程，**地址空间完全独立**。

- **进程切换：**

  1、保存当前进程的 **PCB、寄存器上下文**

  2、**切换页表（Page Table）**

  3、**清空并刷新 CPU 高速缓存 TLB**

  4、加载新进程的页表、上下文

  重点是：

  **页表切换 + TLB 刷新** 非常非常慢。

  TLB 一失效，CPU 访问内存速度直接掉一个数量级。

- **线程切换：**

  同一个进程内的多个线程：

  - **共用页表**
  - **共用虚拟地址空间**
  - **共用打开的文件、信号、内存资源**

  线程切换只需要做三件事：

  1. 保存当前线程的 **栈、寄存器、程序计数器 PC**
  2. 加载下一个线程的 **栈、寄存器、PC**
  3. 直接继续执行

  **完全不用换页表，不用刷新 TLB！**



#### 4、读时共享写时复制

```
1、在fork的时候，内核不会直接将整个父进程的地址空间复制下来，而是先让父子进程共享一个地址空间。只有在需要写入的时候才会复制地址空间，从而使每个进程拥有自己的地址空间

2、作用：
	可以提高fork的效率：不立刻复制整个地址空间，只共享页表，可以提高fork的效率。
	可以减少内存的开销：fork之后可能会exec执行新的程序，这样会造成没有必要的内存开销
	还可以保证进程的隔离：在写入时进行复制，让每个进程都拥有自己的地址空间

```





```c++
1、进程是资源分配的基本单位，资源完全隔离，保证程序安全
2、线程是CPU调度的基本单位，共享进程的核心资源，提升执行效率
3、进程不共享资源：为了隔离，保证程序独立运行、互不干扰
4、线程共享资源：为了高效，降低调度开销，简化线程间协作
大白话：设计目的不同
```





### 三、进程间通信

因为每个进程的地址空间都是独立的，一般是不能互相访问的，但每个进程都共享同一个内核空间，所以进程间的通信都需要通过**内核**

#### 1、管道

管道本质是**内核提供的内存缓冲区**

```
从管道的一端写入的数据，实际上是缓存在内核中的，另一端读取，也就是从内核中读取数据。
```

- **无名管道**

  - 没有名字，只存在于内存，且只能用于父子进程（依靠fork继承的文件描述符）

- **有名管道**

  - 有文件名，存在于文件系统，在不相关的进程也能进行通信

    ```
    有名管道通过文件系统创建一个管道文件作为通信标识，不相关的进程可以通过打开同一个管道文件，由内核将它们关联到同一个内存缓冲区，从而实现数据的传输
    ```

**优点：**简单、轻量、自带同步、自动销毁、适合流式日志与 Shell 流水线。

**缺点：**单向、字节流粘包、缓冲区小、不能削峰解耦、耦合强、无消息分类。

**应用场景：**因为**读空阻塞**和**写满阻塞**天然的同步机制，管道适用于简单的日志传输、Shell脚本流水线这类靠**阻塞同步**稳数据流的场景。

#### 2、消息队列

消息队列是保存在内核中的**消息链表**，在发送数据时，会分成一个一个**独立的**数据单元，也就是消息块。

然后消息队列生命周期随内核，如果没有释放消息队列或关闭操作系统，消息队列会一直存在，但如果进程从消息队列读取了消息体，内核就会把这个消息体删除。

**优点：**解耦、削峰、消息边界、可分类、独立生命周期**。**

**缺点：**有拷贝开销、速度一般、有容量限制、需手动释放、不适合大数据与极高实时性场景。

**适用场景：**解耦生成者和消费者，削峰填谷（如日志收集）。



#### 3、共享内存

共享内存就是将一块虚拟地址映射到相同的物理内存中，这样写入的东西另一个进程可以马上看到，这样就实现了零拷贝，大大提高了通信速度。

![image-20260406212419244](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260406212419244.png)

**优点：**速度快，性能高，适合大数据、高频交互，内存高效节省

**缺点：**无同步机制，不安全。需要创建、映射、同步、卸载、删除，使用复杂。不能解耦、不能削峰。不会自动删除，易泄漏。

**适用场景：**高频、大数据量、低延迟、多进程共享同一份数据的高性能场景



#### 4、信号量

用来控制多进程/线程并发访问共享资源的同步工具，本质是一个计数器 + 等待队列。

两种原子操作：

- P**操作：**将信号量减一，相减后如果信号量＜0，表示资源已经被占用，进程阻塞等待；反之表示资源可使用。
- V**操作：**将信号量加一，相加后如果信号量<=0，则表示当前有阻塞中的进程，于是会将该进程唤醒运行，反之表示没有进程阻塞。



#### 5、信号

### 四、锁

锁是用来**保护共享资源**，保证同一时间只有一个线程/进程访问，避免数据竞争

#### 1、互斥锁

同一时间只允许一个线程持有，其它线程**阻塞**等待，并释放CPU让给其它线程，不占用CPU。



对于互斥锁加锁失败的现象，是由**操作系统内核**实现的。当加锁失败时，内核会将其设置为睡眠状态，等到合适的时机在将其唤醒。所以，互斥锁加锁失败时，会从**用户态**到**内核态**，让内核帮我们切换线程，存在一定的开销成本。

**两次线程上下文切换的成本：**内核将线程的状态从运行设置为睡眠然后睡眠到就绪。

上下文切换的时间可能会比你锁住代码执行的时间还要长，这时就可以使用自旋锁。

#### 2、自旋锁

获取已被占有的锁时，会不断循环等待，不阻塞一直占用CPU。



自旋锁是通过CPU提供的CAS（Compare And Swap）函数，在用户态完成加锁和解锁操作，不会主动产生线程上下文切换，所以相比互斥锁会快一些，开销也小一些。

#### 3、读写锁

读写锁是一种特殊的自旋锁，它把对共享资源的访问者划分为读者和写者，读写互斥，读读共享。

#### 4、悲观锁

互斥锁、自旋锁、读写锁都属于悲观锁。



悲观锁做事比较悲观，它认为多线程同时修改共享资源的概率比较高，于是很容易出现冲突，所以访问共享资源前，先要上锁。

#### 5、乐观锁

乐观锁做事比较乐观，所以也被称为无锁编程，它假定冲突的概率很低，工作方式：先修改完共享资源，再验证这段时间内有没有发生冲突，如果没有其他线程在修改资源，那么操作完成，反之，放弃本次操作。

#### 6、死锁

死锁需要同时满足四个条件才会发生：

**1、互斥条件：**多个线程不能同时使用同一个资源（资源是独占的，不能共享）。

**2、持有并等待条件：**已经拥有资源，又想获取别的资源，且该资源已经被其他线程获取。

**3、不可剥夺条件：**当前资源在自己使用完之前不能被其他线程获取（资源不能被强行夺走）。

**4、环路等待条件：**两个线程的资源获取顺序构成环形链。

#### 3、如何优化文件传输的性能？

**如何减少上下文切换次数：**

读取数据时之所以会发生上下文切换，这是因为用户空间没有权限操作磁盘或网卡，但内核的权限最高，所以要想操作磁盘或网卡我们就需要使用系统提供的系统调用。而一次的系统调用必然会发生两次上下文的切换。**所以要想减少上下文切换次数，就要减少系统调用次数。**



**如何减少数据拷贝次数：**

在数据传输过程会经历4次数据拷贝，而从内核缓冲区到用户的缓冲区和从用户的缓冲区到socket的缓冲区这个过程是没必要存在的，因为它们并没有对数据进行修改。

# 4、线程池

**线程池作用：**

- **复用线程：**避免频繁创建/销毁线程的高开销
- **控制数量：**防止线程过多导致的CPU/内存资源耗尽

```c++
//.h

#ifndef THRD_H
#define THRD_H

#include <thread>
#include <queue>
#include <functional>
#include <vector>
#include <mutex>
#include <atomic>
#include <condition_variable>

class thrd
{
public:
    thrd(size_t core_threads = 4);
    ~thrd();

    template<class F>
    void submit(F&& f){
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            if (stop) {
                throw std::runtime_error("submit task to stopped thread pool");
            }
            tasks.emplace(std::forward<F>(f));
        }
        condition.notify_one();
    }

private:
    void worker();

    std::queue<std::function<void()>> tasks;    //任务列表
    std::mutex queue_mutex; //队列互斥锁
    std::vector<std::thread> workers; //线程列表
    std::condition_variable condition;  //条件变量
    std::atomic<bool> stop;	   //原子布尔变量（原子操作，不可被打断）
};

#endif
```

```c++
//.cpp

#include "thrd.h"

thrd::thrd(size_t core_threads) : stop(false){
    for (size_t i = 0; i < core_threads; i ++){
        workers.emplace_back(&thrd::worker, this);
    }
}

thrd::~thrd(){
    {
        std::unique_lock<std::mutex> lock(queue_mutex);
        stop = true;
    }

    condition.notify_all();

    for (std::thread& worker : workers){
        worker.join();
    }
}

void thrd::worker() {
    //线程不断循环，执行完任务就回到休眠等待队列
    while (true) {
        // 1. 加锁，等待任务/停止信号
        std::function<void()> task;
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            // 条件变量：队列空且未停止时，线程等待
            //返回true就退出阻塞，false阻塞
            condition.wait(lock, [this] { 
                return stop || !tasks.empty(); 
            });
            
            //.wait(lock, predicate)内部逻辑:
            // while(!predicate()){
            //     condition.wait(lock);
            // }
            
            // 若线程池停止且队列为空，退出线程
            if (stop && tasks.empty()) {
                return;
            }
            // 2. 取出队列中的第一个任务
            task = std::move(tasks.front());
            tasks.pop();
        }
        // 3. 解锁后执行任务（避免持有锁执行任务，影响其他线程）
        task();
    }
}
```





# 5、内存池

内存池是一种**内存管理技术**，它通过一次性向操作系统申请一大块内存，然后将其划分为多个小块，通过 **free list** 管理，从而实现 **O(1)** 的内存分配和释放。
 这样可以**减少 malloc/free 系统调用**、**降低锁竞争并减少内存碎片**，在高性能服务器和游戏引擎中被广泛使用。

### 一、为什么需要内存池

普通内存分配流程：

```
new / malloc
   ↓
操作系统或运行时分配内存
   ↓
返回指针
```

释放：

```
delete / free
   ↓
归还给系统
```

问题：

**1、频繁系统调用（慢）：**

系统内存管理涉及 **用户态 → 内核态**和 **复杂的堆管理算法** 如果高频调用malloc/free或new/delete性能会明显下降

**2、内存碎片化，虽然有空闲，但无法满足大块申请**

**3、对象生命周期短**

### **二、内存池核心思想**

**一次申请大块内存，然后自己管理。**

```
系统申请
   ↓
+-----------------------+
|      Memory Pool      |
+-----------------------+

自己切割
```

之后：

```
申请内存 → 从池子里取
释放内存 → 放回池子
```

而不是还给操作系统

# 6、计算机网络

### 	一、传输层

**TCP是面向连接、可靠、字节流的传输层通信协议。**

**核心特点：**

- **面向连接：**通信前必须先建立连接（三次握手），结束要断开（四次挥手）。

- **可靠传输：**丢包重传、按序到达、去重、校验和。

- **字节流：**数据像水流一样连续传输，无边界，需要应用层自己分包。



#### 1、三次握手

简单来说：

```css
客户端先发送SYN，随机初始序列号，然后客户端同意连接，发送SYN+ACK，最后客户端在发送ACK包进行确认。
```

从内核底层来说：

```css
服务器会通过listen监听，申请并初始化半连接队列和全连接队列；然后客户端connect连接时会将状态设置为syn_sent，之后选择本地端口，发出syn握手请求，并启动重传定时器；然后，服务器会对连接队列溢出做出判断，发出syn和ack包，进入半连接队列，并启动定时器；客户端这边再清除重传定时器，发送ack确认；之后，服务器创建新socket，从半连接队列删除，加入全连接队列；最后accept的话会从全连接队列中取走socket。
```



##### 什么是SYN攻击？如何避免？

攻击者短时间伪造不同IP地址的SYN报文，服务器每收到一个SYN报文就会进入SYN_RCVD状态，但服务端发送出去的ACK+SYN无法得到未知IP主机的ACK应答，久而久之就会占满服务端的半连接队列，使服务器无法为正常用户服务。





#### 2、四次挥手

```css
TCP 是全双工协议，断开连接需要双方各自关闭发送通道，所以需要四次挥手。主动关闭方发送 FIN，对方回复 ACK；等对方数据发完后再发送 FIN，最后主动方回复 ACK 并进入 TIME_WAIT 等待 2MSL（2倍最大生存时间），确保报文可靠消失，避免端口复用出错。
```





#### 3、三次握手 + 四次挥手

TCP 是面向连接的可靠传输协议，建立连接用**三次握手**，断开连接用**四次挥手**。

**三次握手的目的**，是为了确认双方的**收发能力都正常**，并同步初始序列号。

1. 客户端向服务端发送 SYN 报文，请求建立连接；

2. 服务端收到后，回复 SYN+ACK，同意连接并确认客户端的请求；

3. 客户端再回复 ACK，确认收到服务端的应答。

   

   至此双方都确认收发正常，连接建立，可以开始传输数据。

**四次挥手的目的**，是因为 TCP 是全双工通信，双方需要**各自独立关闭发送通道**，保证数据传输完整。

1. 主动关闭方发送 FIN，表示自己没有数据要发了；
2. 被动关闭方回复 ACK，表示收到关闭请求，此时只是单向关闭；
3. 等被动关闭方数据发送完毕，再发送 FIN，请求关闭它的发送通道；
4. 主动关闭方回复 ACK，并进入 TIME_WAIT 状态等待 2MSL，确保最后一个 ACK 被对方收到，防止旧报文干扰新连接。

简单总结：

**三次握手是为了安全建立连接，四次挥手是为了优雅断开连接。**



#### 4、重传机制

```css
重传机制是网络通信中，自动补发丢失/损坏的数据包、保证可靠传输的核心技术。
```



四种重传机制：

- **超时重传：**发送数据时，会设定一个定时器，超过一定的时间没接收到对方的ACK确认应答报文，就会重发数据。

  - **引发重传的情况：**数据包丢失、确认应答丢失。

    ![image-20260418153156062](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260418153156062.png)

  - **超时时间设定：**超时重传时间RTO的值应略大于报文往返RTT的值

    - 当RTO较大时，重发效率慢，性能差。
    - 当RTO较小时，重发速度快，可能会导致没有丢失就重发，且会增加网络拥堵，导致更多超时。

    ![image-20260418153726778](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260418153726778.png)

  - 如果重发的数据，再次超时又需要重传时，TCP的策略是**超时间隔加倍**。两次超时说明网络环境差，不宜频繁发送。

- **快速重传（解决超时重发的时间等待）：**不以时间为驱动，而是以数据为驱动进行重传。

  - 当收到**3个相同的ACK报文**时，才会在定时器超时前，重发丢失的数据。

    ![image-20260418155234717](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260418155234717.png)

  - **局限性：**必须收到3次相同的ACK才会触发，如果丢包后，后续没有任何数据进行传输，就不会有重复的ACK，快速重传就会失效，就只能依靠超时重传了。

- **SACK方法（解决快速重传问题）：**在TCP头部【选项】字段里加上一个SACK的东西，它可以将已收到的数据信息发送给【发送方】，这样发送方就可以知道丢失的数据包并进行重传。

- **D-SACK:**主要用于使用SACK来告诉发送方有哪些数据被重复接收了。

#### 7、机制串讲

TCP 的目标是在不可靠的网络上，实现**可靠、高效、公平**的传输。这一切都建立在**滑动窗口**这个基础框架上。

首先，**滑动窗口** 是核心骨架。它通过左右边界限定了发送方可以连续发送的序号范围，并随着 ACK 不断向右滑动，实现批量数据传输，这是保证高效传输的基础。

在此之上，有两套控制机制：

第一是**流量控制**。它是端到端的控制，为了防止发送方发得太快把接收方缓冲区撑爆。它依靠接收窗口 **rwnd** 实现，发送方必须遵守 `发送窗口 ≤ rwnd`。

第二是**拥塞控制**。它是面向网络的控制，为了防止路由器队列溢出导致网络崩溃。它依靠拥塞窗口 **cwnd** 来感知网络状况，通过**慢启动**和**拥塞避免**阶段来试探并平稳提升速率。

最终，发送窗口的大小是由 **min(rwnd, cwnd)** 决定的，既保证接收方受得了，也保证网络不被堵死。

最后，为了保证**可靠性**，TCP 配备了重传机制。当超时未收到 ACK，会触发**超时重传**；而当收到**3 个重复 ACK**时，会触发**快速重传**，在定时器过期前提前重传丢失的数据，并结合 **SACK** 实现精准重传，快速恢复。

总结一下：**滑动窗口是骨架，流量和拥塞控制是两翼，重传机制是保险**，四者协同，共同保证了 TCP 的稳定运行。

##### 常见状态码：

`1xx` 类状态码属于**提示信息**，是协议处理中的一种中间状态，实际用到的比较少。

`2xx` 类状态码表示服务器**成功**处理了客户端的请求，也是我们最愿意看到的状态。

- **200：**表示一切正常。如果是非**HEAD**请求，服务器返回的响应头都会有body数据。
- **201：**创建资源成功（POST/PUT新建时常用）
- **204：**和200基本相同，但响应头没有body数据（常用于删除接口）。

`3xx` 类状态码表示客户端请求的资源发生了变动，需要客户端用新的URL重新发送请求获取资源，也就是**重定向**。

- **301：**表示永久重定向，说明请求的资源已经不存在了，需要改用新的URL再次访问。
- **302：**表示临时重定向，说明请求的资源还在，但暂时需要用另一个URL来访问。

301和302都会在响应头里使用**Location**，指明后续要跳转的URL，浏览器会自动重定向新的URL。

- **304：**表示资源未修改，重定向已存在的缓冲文件，也就是缓存重定向，也就是告诉客户端可以继续使用缓存资源，用于缓存控制。

`4xx` 类状态码表示客户端发送的**报文有误**，服务器无法处理。

- **400：**表示客户端请求的报文有错误。
- **403：**表示服务器禁止访问资源。
- **404：**表示请求的资源在服务器上不存在或未找到。

`5xx` 类状态码表示客户端请求报文正确，但是服务器处理时内部出现错误。

- **500：**服务器代码报错或异常
- **501：**表示服务器自身工作正常，访问后端服务器发送了错误，通常是服务器作为网关或代理时返回的错误码。
- **503：**表示服务器繁忙，暂时无法响应客服端。



##### 常见字段：

- **Host：**客户端发送请求时，用来指定服务器的域名。
- **Content-Length：**表明本次回应的数据长度。
- **HTTP粘包解决：**HTTP 协议通过设置回车符、换行符作为 HTTP header 的边界，通过 Content-Length 字段作为 HTTP body 的边界。
- **Connection：**最常用于客户端要求服务器使用「HTTP 长连接」机制，以便其他请求复用。HTTP 长连接的特点是，只要任意一端没有明确提出断开连接，则保持 TCP 连接状态。
- **Content-Type：**Content-Type 字段用于服务器回应时，告诉客户端，本次数据是什么格式。
- **Content-Encoding：**说明数据的压缩方法。表示服务器返回的数据使用了什么压缩格式。



##### 常见请求：

- **GET：**从服务器获取指定的资源。
- **POST：**根据请求负荷（报文body）对指定资源做出处理。
- **HEAD：**只拿响应头，不拿内容。
- **PUT：**全量跟新资源。
- **PATCH：**局部更新。
- **DELETE：**删除资源。
- **GET和POST区别：**
  - GET 用于获取资源，是只读、安全、幂等、可缓存，参数在 URL。
  - POST 用于提交处理，是写操作、不安全、不幂等、不缓存，参数在 Body。



##### 缓存技术：

```css

```

- **强缓存：**浏览器直接用本地缓存，不发请求道服务端。
- **协商缓存：**就是与服务端协商之后，通过协商结果来判断是否使用本地缓存。
  - 返回**304**，继续使用旧缓存。
  - 返回**200** + 新资源

#### 2、HTTPS

```css
HTTPS = HTTP + TLS/SSL，在应用层和传输层之间加了一层安全协议，默认端口为443，解决HTTP明文传输不安全的问题。
```

![image-20260422203444978](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260422203444978.png)





##### 工作流程：

**TCP 三次握手**建立连接
**TLS 握手**（HTTPS 核心）

- 客户端发送支持的加密套件、随机数 C。

- 服务器返回选中的加密套件、随机数 S和服务器的**数字证书。**

- 客户端验证证书（校验 CA、域名、有效期），生成**预主密钥 Pre-Master Secret**，用**服务器公钥加密**发给服务器。

- 双方用 C + S + Pre-Master 计算出**相同的对称会话密钥**，后续所有 HTTP 数据都用**对称加密**传输。

  ```
  客户端和服务器先打招呼，交换随机数，服务器下发证书，客户端验证后生成预主密钥并用公钥加密发给服务器，双方算出会话密钥，之后用对称加密通信。
  ```

传输完毕，TCP 断开



##### TLS（传输层安全）作用：

1、**信息加密**：通过**混合加密**的方式可以保证信息的**机密性**，解决了窃听的风险。

HTTPS采用的是**对称加密**和**非对称加密**结合的「混合加密」方式:

- 握手阶段用**非对称加密**安全交换会话密钥，解决密钥交换问题；
- 传输数据阶段用**对称加密**加密业务数据，解决性能问题，兼顾安全和效率。

采用「混合加密」的方式的原因:

- **对称加密**只使用一个密钥,运算速度快,密钥必须保密,无法做到安全的密钥交换。

- **非对称加密**使用两个密钥:公钥和私钥,公钥可以任意分发而私钥保密,解决了密钥交换问题但速度慢。

  ![image-20260422210155322](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260422210155322.png)

2、**完整性校验**：通过**摘要算法**的方式来实现完整性，防止数据被篡改。

发送方计算数据摘要，和数据一起发给对方；接收方通过同样的算法计算摘要和接收到的进行对比。

![image-20260422210854612](C:\Users\manong\AppData\Roaming\Typora\typora-user-images\image-20260422210854612.png)

3、**校验机制**：将服务器公钥放入到**数字证书**中，解决了冒充的风险。

服务器用私钥对数据的摘要进行签名，客户端收到后，用公钥验证签名，确保数据确实来自服务器，且没被篡改。





#### 3、HTTP与HTTPS区别

- HTTP是超文本传输协议,信息是明文传输,存在安全风险的问题。HTTPS则解决HTTP不安全的缺陷,在TCP和HTTP网络层之间加入了SSL/TLS安全协议,使得报文能够加密传输。 
- HTTP 连接建立相对简单，TCP 三次握手之后便可进行 HTTP 的报文传输。而 HTTPS 在 TCP 三次握手之后,还需进行SSL/TLS的握手过程,才可进入加密报文传输。
- 两者的默认端口不一样, HTTP默认端口号是80, HTTPS默认端口号是443。 
- HTTPS协议需要向CA (证书权威机构)申请数字证书,来保证服务器的身份是可信的。

### 4、面经

#### 1、性能监控项目，解析/proc文件下meminfo去获取内存的一些使用情况，说说这里面有哪些资源的一些使用参数。

**容量与使用率：**

- **MemTotal：**总内存
- **MemFree/MemAvailable：**空闲与可用
- **使用率：**我们用 `(Total - Available) / Total`，比单独用 Free 合理。
  - 单纯的Free仅代表完全未使用的裸内存，Linux会大量利用空闲内存做页缓存和slab可回收缓存，导致Free很低，无法反映真实水位。

**缓存与可回收资源：**

- **Buffers、Cached：**块缓存与页缓存
- **SReclaimable、KReclaimable：**可回收slab/可回收内核内存，和内存压力时可释放能力有关。

- 使用率：我们用 `(Total - Available) / Total`，比单独用 Free 合理。

**页的生命周期（冷热页）：**

- **Active/Inactive**
- `Active(file)`、`Inactive(file)`、`Active(anon)`、`Inactive(anon)`：区分文件缓存与匿名内存，便于判断是 IO 缓存吃满还是进程堆内存吃满。

**回写与风险信号：**

- **Dirty/Writeback：**待落盘的脏页和正在刷盘回写的页。

**交换空间总量和已交换空间：**SwapTotal、SwapFree



#### 2、仅仅是做了一个性能的采集吗，有没有参与一些性能的优化，比如内存优化呀？

**内存优化 ：**
 通过 `mmap` 实现内核态与用户态的零拷贝数据传递，避免传统 read/write 系统调用的数据拷贝开销；在 CPU 状态监控中使用 `std::unordered_map` 缓存历史数据，通过差分计算减少重复计算；内核模块采用 `__get_free_pages`分配物理页对齐的共享内存，提升内存访问效率。

**性能优化 ：**
使用内核模块直接采集数据，避免用户态频繁系统调用；采用 `hrtimer` 高精度定时    器实现毫秒级定时更新；通过 eBPF 在内核态过滤网络数据，减少用户态数据处理量；设计缓存机制存储上次采样值，通过差值计算使用率变化率，避免瞬时波动干扰分析。



#### 3、有没有通过一些渠道去考虑或者了解，比如像CPU负载过高，内存可用比较少，这些情况我该怎么去优化？

1、CPU负载高要先分清**负载和利用率**，平均负载高不代表CPU利用率高，可能是大量进程排队或卡在IO。我会先看CPU总使用率和分项：**用户态、内核态、iowait、中断和软中断**，再结合磁盘、网络来判断是**算力瓶颈**还是**IO瓶颈**。

如果是**CPU瓶颈：**用top找热点进程和函数，从算法、数据结构、锁竞争、线程模型上进行优化。

如果是**IO瓶颈：**重点优化磁盘读写、缓存和异步IO。

最后如果单机已到上限，用水平扩展、异步化、缓存分担压力。

#### 4、linux监控项目，说说使用ebpf进行网络流量统计的流程，ebpf在网络协议栈里面如何工作？

首先通过libbpf打开并加载eBPF字节码，为每个网卡创建Hook挂载点，然后将eBPF程序分别附加到ingress和egress两个方向。当数据包从网卡进出协议栈时，会触发ingress Hook和egress Hook，eBPF程序从skb结构中获取**网卡索引**和**数据包长度**，通过原子操作更新BPF Map中的接收统计。用户态程序通过遍历BPF Map获取各网卡的累计数据，结合时间戳计算每秒速率。



#### 5、性能监控项目，读取到了哪些内存指标，读取到之后如何去做一个分析？

首先从整体水位入手，观察**总内存**、**可用内存**、**缓存及使用率**，结合指标变化率识别突增趋势，提前预警风险。接着通过**Swap**指标判断交换压力——若Swap持续升高且频繁换入换出，说明物理内存不足，依赖磁盘交换拖慢性能。然后分析**脏页与回写页**，若积压过高表明磁盘IO跟不上，引发系统卡顿。再通过**匿名页**观测业务内存开销，若匿名页持续上涨而映射页无变化，可排除缓存干扰，判定存在内存泄漏。最后通过**Slab内核内存**区分可回收与不可回收内存，若不可回收Slab持续走高，说明内核侧内存占用异常。



#### 6、CPU负载如何去做一个分析，怎么判断具体系统是哪里的问题？

**CPU利用率：**CPU正在干活的时间占比。

**负载：**一段时间内CPU运行队列里等待的进程数+不可中断睡眠进程数量（1/5/15）分钟。

我会同时看负载，以及通过内核模块拿到CPU的使用情况去进行分析：

- 如果**负载高+CPU利用率高**，说明是用户态或内核态计算密集导致，可以通过top、pidstat、perf定位高CPU进程和线程，一般是因为死循环、不合理轮询或大量计算。

- 如果是**负载高但CPU利用率低**，基本是因为IO阻塞造成的，大量进程处于D不可中断状态。可以通过iostat、iotop检查磁盘IO，同时排查内存Swap、数据库慢查询、网络超时等问题。
- 另外还有**软中断**的统计（包括**高优先级和轮询类**），如果这部分异常，一般会结合**网络吞吐**，因为我还用了**eBPF在协议栈统计网卡收发**，判断是不是**网络路径压力**把中断顶高了。
- 最后所以关键的指标都会算**变化率**，用来**提前发现徒变、做预警**。

- Netfilter ：贯穿整个协议栈，功能强大但性能开销较大

#### 18、本来可以用proc方式获取数据，为什么要用内核模块？

在我们的性能监控系统里，**没有把 /proc 作为首选采集方案**，主要是出于性能、精度和扩展性的考虑。
首先，读 /proc 需要频繁的系统调用和上下文切换，每次读取都要陷入内核态、再拷贝数据到用户态，高频采集下开销很大，会影响业务性能；而我们的内核模块 + mmap 方案，可以直接在内核态读取数据，再通过共享内存映射到用户态，实现零拷贝、无系统调用，性能要高得多。
其次，/proc 的数据大多是内核按固定周期更新的快照，时效性和精度有限，而内核模块我们使用hrtimer高精度定时器实现毫秒级的数据读取，更适合我们做高精度监控。
另外，/proc 是文本格式，解析逻辑复杂，而且不同内核版本的格式还可能变化，维护成本高；内核态采集直接读取结构体，数据是结构化的，扩展性更好。
所以我们把 /proc 作为降级方案，当内核模块无法加载时，自动切换到 /proc 读取，保证各种环境下监控都能稳定运行。

#### 21、分布式linux性能监控这块，内核模块是有做一个mmap零拷贝，零拷贝这块你是怎么理解的啊？

零拷贝是一种高效的数据传输技术，核心思想是避免数据在内存中进行不必要的拷贝操作。

在传统的数据读取方式中，比如读取`/proc`文件，数据需要经过多次拷贝：首先从内核空间拷贝到内核缓冲区，再从内核缓冲区拷贝到用户空间缓冲区，这个过程会产生两次数据拷贝和两次用户态/内核态切换的开销。

而通过mmap实现零拷贝，本质是将内核空间的一块内存区域直接映射到用户空间。这样用户程序就可以像访问自己的内存一样直接访问内核数据，不需要进行数据拷贝。在项目中，内核模块先在内核空间分配共享内存区域，然后通过设备文件的mmap接口将这块内存映射到用户进程的地址空间。用户程序读取数据时，直接从映射的内存区域读取，避免了传统方式中的数据拷贝开销。

这种方式的优势非常明显：首先是性能提升，减少了数据拷贝和系统调用次数；其次是内存效率高，不需要额外的缓冲区存储数据；最后是实时性好，内核数据更新后用户程序能立即看到最新数据。在性能监控系统中，这种设计保证了数据采集的高效性和实时性。



#### 22、那常用的一些零拷贝的框架，比如说市面上比较成熟的像dpdk这种用过吗？有听说过吗？

是的，我听说过DPDK（Data Plane Development Kit），它是Intel推出的高性能数据包处理框架。

DPDK主要特点是通过用户态轮询模式（Poll Mode Driver）绕过内核协议栈，直接访问网卡硬件，实现线速数据包处理。相比传统方式，它减少了内核态切换和数据拷贝开销，在网络数据包处理性能上有显著优势，常用于NFV、SDN等高性能网络场景。

不过在项目中，我们选择了eBPF和内核模块的方案，主要是考虑到：

1. 复杂度 ：DPDK需要特殊的网卡驱动和环境配置，部署复杂度较高；而eBPF和内核模块对现有系统的侵入性较小。
2. 功能需求 ：项目主要是性能监控，不需要线速数据包处理能力，eBPF和内核模块已经能满足需求。
3. 生态兼容性 ：eBPF是Linux内核原生支持的技术，与现有系统工具链兼容性更好。



#### 23、讲讲用c语言写的部分---内核模块

项目中用C语言编写的内核模块主要负责CPU状态和软中断的采集，核心实现包括以下几个部分：

首先是模块结构设计，采用标准的Linux内核模块结构，通过 module_init 和 module_exit 宏分别定义初始化和卸载函数。在初始化阶段完成设备号分配、字符设备注册、共享内存分配和高精度定时器启动；卸载时反向释放所有资源。

然后是数据采集机制，通过 per_cpu(kstat_cpu) 宏访问每个CPU的统计数据结构，获取用户态、系统态、空闲时间、IO等待等指标，使用 hrtimer 高精度定时器实现每秒定时更新，确保数据的实时性。

零拷贝数据传输是关键优化点，在内核空间使用 __get_free_pages 分配物理页对齐的内存区域，通过字符设备的 mmap 接口映射到用户空间，用户程序可以直接访问内核数据，避免了传统系统调用的数据拷贝开销。

字符设备实现方面，注册了 /dev/cpu_stat_monitor 设备，实现 open 、 release 和 mmap 操作，其中 mmap 函数通过 remap_pfn_range 将内核物理页映射到用户进程地址空间，并设置页面属性为不可缓存以保证数据一致性。

整个内核模块设计确保了高效、实时的数据采集，为性能监控系统提供了准确的底层数据支持。



#### 24、cpu信息统计，内核模块是读取文件还是直接写的吗？

内核模块是直接访问内核数据结构，而不是读取文件。

在项目中，我编写的内核模块通过 per_cpu(kstat_cpu) 宏直接访问Linux内核中每个CPU的统计数据结构，获取用户态、系统态、空闲时间、IO等待等原始数据。这些数据是内核在运行过程中实时维护的，不需要读取 /proc 等文件系统。

模块使用 hrtimer 高精度定时器每秒更新一次数据，并将采集到的信息写入通过 __get_free_pages 分配的共享内存区域。用户态程序通过 mmap 将这块内存映射到自己的地址空间，直接读取数据，实现了零拷贝的数据传输。

这种方式相比读取 /proc/stat 等文件，避免了频繁的文件操作和文本解析开销，采集效率更高，数据也更准确。当然项目也保留了降级方案，当内核模块不可用时会自动切换到读取 /proc 文件。



#### 25、怎么分析这些数据呢？是只通过内存变化就推出吗？

数据分析不是只看内存变化，而是通过多维度指标的综合分析来判断系统状态。

在项目中，我主要从几个方面进行分析：首先是指标对比分析，采集到的数据会与历史数据对比，比如CPU使用率通过缓存上次采样值计算变化率，判断负载趋势；内存方面不仅看MemFree，更关注MemAvailable，因为它考虑了可回收缓存页。

其次是多维度关联分析，将CPU、内存、网络、磁盘等指标结合起来，比如CPU负载升高时同时查看IO等待时间，如果iowait很高说明可能是磁盘IO瓶颈导致CPU等待。

然后是阈值告警机制，设置合理阈值，当指标超过阈值时触发告警。还有趋势预测，通过历史数据分析预测系统未来性能趋势，比如内存使用持续增长可能预示内存泄漏。

最后是异常模式识别，比如CPU软中断中net_rx突然激增可能意味着网络攻击或异常流量。通过这些综合分析手段，能够全面了解系统运行状态，及时发现性能瓶颈和潜在问题。

#### 28、如果想实现一个最简单的驱动，加载驱动时打印“驱动加载”信息，退出驱动时就打印“驱动卸载”信息，都需要用到哪些函数啊？描述下

**实现最简单的Linux驱动需要用到以下几个关键函数：**

**1. `module_init()` 宏**  
用于注册模块加载时的初始化函数。当驱动被加载（如执行 `insmod` 命令）时，该宏指定的函数会被调用。

**2. `module_exit()` 宏**  
用于注册模块卸载时的清理函数。当驱动被卸载（如执行 `rmmod` 命令）时，该宏指定的函数会被调用。

**3. `printk()` 函数**  
内核态的打印函数，用于在系统日志中输出信息。与用户态的 `printf` 类似，但可以指定日志级别（如 `KERN_INFO`）。

**4. `MODULE_LICENSE()` 宏**  
声明模块的许可证（如 `GPL`），这是内核模块的必要声明。

**示例代码结构：**

```c
#include <linux/module.h>

static int __init hello_init(void) {
    printk(KERN_INFO "驱动加载\n");
    return 0;
}

static void __exit hello_exit(void) {
    printk(KERN_INFO "驱动卸载\n");
}

module_init(hello_init);
module_exit(hello_exit);
MODULE_LICENSE("GPL");
```

**工作流程：**  

- 加载驱动时，`hello_init()` 被调用，输出"驱动加载"
- 卸载驱动时，`hello_exit()` 被调用，输出"驱动卸载"



#### 29、protobuf？服务端和客户端的信息如何对齐？编码解码的过程如何对齐？

**面试连贯回答**：

Protobuf是一种高效的二进制序列化协议，服务端和客户端通过统一的`.proto`文件定义来保证信息对齐。

**信息对齐方式**：
首先，服务端和客户端需要共享同一个`.proto`文件，这个文件定义了数据结构和接口。通过protoc编译器可以生成不同语言的代码，比如服务端用C++，客户端用Python，但生成的代码语义完全一致。这样服务端和客户端就拥有了相同的数据结构定义，确保了信息的一致性。

**编码解码过程对齐**：
编码时，发送方根据`.proto`定义将数据结构序列化为二进制字节流，这个过程由生成的代码自动完成。解码时，接收方同样根据`.proto`定义将二进制字节流反序列化为对应的数据结构。由于双方使用相同的`.proto`文件和相同的序列化规则，编码解码过程自然对齐。

**关键保证**：
Protobuf使用Tag-Length-Value格式，每个字段都有唯一的编号，即使协议升级添加新字段，旧版本也能兼容处理，保证了良好的向后兼容性。在我们的项目中，通过gRPC和Protobuf实现Manager和Worker之间的通信，确保了分布式系统中各节点的数据一致性和通信效率。



#### 30、grpc？编码解码的过程在哪里进行？

在gRPC中，编码解码过程主要发生在客户端和服务端的**Stub层**以及框架内部的**协议层**。

客户端调用远程方法时，首先在**Stub层**将业务数据结构序列化为Protobuf二进制格式，然后**gRPC框架**将其封装成HTTP/2的DATA帧通过网络发送。服务端接收到数据后，先在协议层解析出Protobuf二进制数据，再在Service实现层反序列化为业务数据结构，最后调用业务逻辑处理。

整个过程由gRPC框架自动完成，开发者只需定义.proto文件并使用protoc生成代码，无需手动处理序列化和网络传输细节。这种设计让开发者能专注于业务逻辑，同时保证了高效的二进制传输和跨语言兼容性。

#### 32、遇到的难点，怎么解决的

在这个分布式监控系统项目中，我遇到的最大技术挑战，是解决NO_HZ （tickless）无时钟中断模式下，CPU 状态统计不准，以及 eBPF 网络监控与内核模块 CPU 监控时间不同步的问题。
现在生产环境的服务器，为了节能都会开启 NO_HZ 模式，CPU 空闲时会停止周期性时钟中断，进入深度睡眠。这就带来两个核心问题：第一，传统依赖时钟中断的 CPU 统计方式失效，/proc/stat 里的空闲时间严重滞后，导致 CPU 利用率、负载计算极不准确。第二，eBPF 网络采集和内核模块 CPU 采集，时间基准不统一，数据无法对齐，监控关联分析完全做不了。
针对这个问题，我设计实现了三层时间校准机制：第一层，在内核模块中，直接使用内核专门针对 NO_HZ 提供的 get_cpu_idle_time_us 接口，获取精确的空闲时间，同时兼容老内核，自动回退到传统统计方式，保证适配性。第二层，在用户态采集程序中，统一使用 steady_clock 单调时钟，避免 NTP 时间跳变带来的误差，保证时间间隔计算准确。第三层，让 eBPF 和内核模块共用内核 CLOCK_MONOTONIC 作为统一时间基准，通过 mmap 共享内存实现零拷贝数据同步，消除跨模块时间偏差。

```css
系统为了降低功耗一般会开启NO_HZ模式，在这个模式下，当CPU空闲时，它是没有时钟中断的，没有时钟中断，它就不会主动的去更新CPU的状态，就会导致系统自带的计时数据滞后、不准确。其次这个项目eBPF网络采集和内核模块CPU采集是同时工作的，但由于它们的时间基准不统一，就会导致数据无法对齐，监控的关联分析就完全做不了。
```

```css

其次我还让eBPF和内核模块共用内核CLOCK_MONOTONIC作为统一的时间基准，并通过mmap共享内存实现零拷贝数据同步，消除跨模块时间的偏差。
```

### 2、为什么需要RAG

- **减少幻觉：**LLM倾向于“编造”不存在的事实。RAG通过提供真实、可靠的外部信息，大大降低了LLM产生不准确或虚假信息的可能性
- **知识时效性：**LLM的训练数据是静态的。RAG允许你使用最新的数据，而无需重新训练或微调LLM
- **可追溯性：**由于回答是基于检索到的文档生成的，你可以很容易地提供引用来源，让用户知道答案来自哪里，增加了透明度和信任度。
- **成本效益：**与耗时且昂贵的 LLM 微调相比，RAG 通常是更经济高效的解决方案。你只需更新向量数据库即可。

eBPF（Extended Berkeley Packet Filter，扩展伯克利包过滤器） 是 Linux 内核里的安全沙盒虚拟机，允许你不修改内核源码、不加载传统内核模块、不重启系统，就能在内核态安全运行自定义程序，实现高性能可观测、网络、安全能力。

Linux 内核里的安全可编程虚拟机，用事件驱动、沙盒安全、JIT 加速、BPF Maps 交互，实现零侵入、高性能的可观测、网络、安全能力，是云原生时代的核心技术底座。
```



| 特性   | 传统内核模块                      | eBPF 程序                                   |
| ------ | --------------------------------- | ------------------------------------------- |
| 安全性 | 直接操作内核内存，崩溃 = 系统宕机 | 验证器 + 沙箱，崩溃不影响内核               |
| 部署   | 编译、加载、重启，内核版本绑定    | 动态加载 / 卸载，无需重启，CO-RE 跨内核兼容 |
| 开发   | 复杂、风险高、调试难              | 工具链完善、安全验证、易调试                |



### 一、工作流程

#### 1. 编写 eBPF 程序（内核态，C 语言为主）

- 事件驱动：挂载到内核**钩子点（Hooks）**，事件触发时自动执行（如网络包到达、系统调用、函数进入 / 退出）。
- 核心限制：**无无限循环、栈大小有限（512 字节）、只能调用内核提供的 Helper 辅助函数、不能直接访问任意内核内存**。
- 示例：跟踪 `execve` 系统调用，记录进程启动信息。

#### 2. 编译为 eBPF 字节码

- 用 Clang/LLVM 编译：`clang -O2 -target bpf -c ebpf_prog.c -o ebpf_prog.o`，生成包含 BPF 指令的目标文件。

#### 3. 加载到内核（用户态程序调用 `bpf()` 系统调用）

- 用户态程序（C/Python/Go）通过 `bpf()` 系统调用，把字节码注入内核。

#### 4. 内核验证（Verifier，安全核心）

- 加载前必须通过静态安全检查：
  1. 所有路径都能正常结束，**禁止死循环**（支持有界循环，必须能被验证器确认会退出）
  2. 无未初始化变量、无内存越界访问
  3. 只调用合法 Helper、只访问合法上下文
  4. 程序大小、复杂度在限制内。
- 不通过则直接拒绝加载，**从根源避免内核崩溃**。

#### 5. JIT 编译 + 挂载执行 + 数据交互

- JIT 即时编译：把通用字节码转为**原生机器码**，执行速度接近内核原生代码。
- 挂载到钩子：绑定到指定事件点（kprobe、tracepoint、XDP、TC、socket 等），事件触发时运行。
- 数据交互：通过 **BPF Maps**（内核态 / 用户态共享的键值存储，哈希、数组、环形缓冲区等）交换数据、保存状态；用户态读取 Map 做展示 / 分析。

### 二、核心组件

#### 1. 钩子（Hooks）：程序在哪里运行

eBPF 程序必须挂载到内核预定义的钩子，决定触发时机与上下文：

- **跟踪类**：kprobe（内核函数动态探针）、kretprobe、uprobe（用户态函数探针）、tracepoint（内核静态跟踪点，稳定）、perf_event（性能事件）
- **网络类**：XDP（eXpress Data Path，网卡驱动层最早处理包，微秒级，DDoS / 负载均衡）、TC（流量控制，网络栈中间层）、socket filter（套接字过滤）、cgroup sock（容器网络控制）
- **安全 / 系统类**：seccomp（系统调用过滤）、LSM（Linux 安全模块，文件 / 进程 / 网络安全）

#### 2. BPF Maps：数据桥梁

- 内核与用户态共享内存的**高效数据结构**，是 eBPF 程序保存状态、传递数据的唯一方式。
- 常用类型：哈希表、数组、per-CPU 数组、环形缓冲区（ringbuf，高性能事件上报）、LRU、栈跟踪等。
- 支持原子操作，多程序 / 多 CPU 安全访问。

#### 3. Helper Functions：内核开放 API

- eBPF 不能直接调用内核函数，只能调用内核提供的**稳定 Helper**：读取上下文、操作 Map、获取时间 / 进程 ID、修改网络包、打印调试信息（bpf_trace_printk）等。

#### 4. CO-RE（Compile Once, Run Everywhere）

- 解决内核版本不兼容问题：用 BTF（BPF Type Format）内核类型信息，编译一次，在不同内核版本上自动重定位数据结构，**真正跨内核运行**，是生产级开发标准。

### 三、三大核心应用场景

#### 1. 可观测性（最广泛）：零侵入、全栈透视

- 无需修改应用代码、无需重启，在内核层直接采集：

  - 系统调用、进程 / 线程创建、文件读写、网络连接
  - 函数调用栈、CPU / 内存 / IO 热点、延迟分布
  - 容器 / 微服务流量、gRPC/HTTP 等应用层协议解析

  

- 工具：bpftrace（快速跟踪脚本）、BCC、Pixie、DeepFlow

#### 2. 高性能网络：替代 iptables，云原生网络底座

- **XDP**：网卡驱动层直接处理包，丢弃 / 转发 / 修改，**微秒级延迟、百万级 PPS**，用于 DDoS 清洗、负载均衡、流量过滤。
- **TC（Traffic Control）**：网络栈中间层，做流量控制、QoS、服务网格、K8s 网络策略（Cilium 核心）。
- 优势：比 iptables 更快、更灵活、支持容器 / 服务感知。

#### 3. 安全：内核级、不可绕过的防护

- 系统调用过滤（seccomp-bpf）：限制容器 / 进程可调用的系统调用，防止提权、恶意行为。
- 运行时检测：监控文件访问、进程执行、网络连接，实时发现异常、入侵、恶意代码。
- 网络微隔离：L3-L7 细粒度策略，比传统防火墙更高效、更适配云原生。

### 四、开发工具链

- **libbpf**：C/C++ 官方库，支持 CO-RE，生产级首选，编译一次到处运行。
- **BCC**：Python 前端，内嵌 eBPF 代码，快速写跟踪脚本、原型开发。
- **bpftrace**：类 awk/DTrace 高级语言，一行命令做系统跟踪，调试神器。
- **Cilium**：基于 eBPF 的 K8s CNI、网络策略、可观测性平台。

### 五、关键技术要点总结

1. **安全基石**：Verifier 验证 + 沙箱执行 + 权限控制（CAP_BPF/root），保证内核稳定。
2. **性能优势**：内核态执行、避免用户态 / 内核态频繁拷贝、JIT 编译、XDP 绕开部分网络栈。
3. **零侵入**：不修改应用、不修改内核、不重启，动态加载 / 卸载。
4. **跨平台兼容**：CO-RE + BTF，解决内核版本碎片化问题。

### 2、面经

#### 1）介绍一下项目

这是一个基于 C++ 实现的 **MCP（Model Context Protocol）协议框架**，核心目标是为 AI 大模型提供标准化的工具调用能力。项目采用 **客户端-服务器架构**，客户端通过 **JSON-RPC 协议** 与服务器通信，支持 **STDIO**（本地进程间通信）和 **SSE**（远程 HTTP 调用）两种传输模式。服务器采用 **插件化设计**，能够动态加载 Windows(.dll)、Linux(.so)、macOS(.dylib) 格式的插件，目前已内置计算器、天气查询、代码审查等多个工具插件。项目的一大亮点是集成了 **RAG-MCP 智能工具检索模块**，通过调用阿里百炼 Embedding API 将工具描述向量化，建立向量索引，当用户查询时先进行语义相似度检索，返回最相关的 Top-K 工具供 LLM 选择，从而避免一次性向大模型发送全量工具导致 Token 消耗过大和准确性下降的问题。

#### 2）项目为什么用的是JSON-RPC？它是怎么设计的？和标准的rpc有什么区别？

之所以选择 JSON-RPC，**一方面**是因为 MCP 协议规范本身就基于 JSON-RPC，**另一方面**是 AI Agent 场景需要动态工具发现能力——工具接口在运行时才能确定，无法用 gRPC 这种需要预先编译 .proto 静态定义的方式。JSON 格式对 AI 大模型也更友好，易于解析和生成文本。

MCP 的 JSON-RPC 设计有几个独特之处：首先是**统一响应格式**，所有工具调用都返回 `content` 数组和 `isError` 标志，让 AI 能以一致的方式处理结果；其次是**插件动态注册机制**，服务器启动时扫描插件目录，将工具实时注册到可用列表；最后是**通知机制**，服务器能通过 SSE 通道主动向客户端推送消息。

与标准 RPC（如 gRPC）相比，核心差异在于：gRPC 以强类型静态契约为核心，编译阶段即可完成类型校验，依托 Protobuf 二进制序列化实现超高传输性能，更适用于接口固定、追求高效通信的微服务场景；JSON-RPC 依靠文档约束类型，接口可运行时动态扩展，牺牲部分性能换取灵活度与可读性，完美契合 AI Agent 插件热更新、工具动态拓展的业务需求。

#### 3）客户端发起一个请求的时候，服务端怎么找到对应的工具？（mcp工具调用的全流程）

当客户端发送 `tools/call` 请求到服务端时，整个路由和查找过程是这样的：服务端启动时通过 `server->OverrideCallback()` 为 `tools/call` 方法注册了一个处理函数，这个函数内部引用了插件加载器 `loader`。当请求到达时，处理函数首先从 JSON 中解析出客户端要调用的工具名称，然后通过一个双层循环去遍历所有已加载的插件和每个插件下注册的工具，逐个比较名称是否匹配。匹配成功后就调用该插件的 `HandleRequest()` 方法，把完整的请求 JSON 字符串传进去，插件内部负责解析参数、执行逻辑、返回结果。服务收到插件返回的字符串后释放插件分配的内存，包装成 JSON-RPC 格式的响应发回给客户端。如果遍历完所有插件都没找到匹配的工具名称，就返回一个包含 `Tool not found` 错误信息的响应。

工具之所以能被找到，是因为服务启动时会调用 `loader->LoadPlugins()` 扫描指定目录下的 `.so` 动态库文件并加载，加载过程中每个插件通过 `GetToolCount()` 和 `GetTool()` 接口把自己拥有的工具暴露出来。`tools/list` 接口的实现也是类似的遍历逻辑，它把所有插件的工具收集起来返回给客户端，这就是为什么客户端调用 `listTools()` 能获得完整工具列表的原因。当前的查找算法是简单的线性遍历，时间复杂度是 O(插件数 × 工具数)，对于插件数量有限的本地场景足够高效，如果未来要支持大规模插件可以考虑用哈希表建立名称索引来优化。

#### 4）假设客户端请求的QPS很高，高并发环境下怎么去优化性能？

在高并发场景下，可以从以下几个层面进行性能优化：

首先是**请求处理层面**，当前服务端采用单线程阻塞模型，请求只能串行处理，无法利用多核CPU。可以引入线程池，将请求提交到线程池中并行处理，这样能显著提升吞吐量。

其次是**连接管理层面**，对于SSE远程调用模式，频繁创建和销毁连接会产生大量开销。可以实现连接池，复用已建立的连接，减少连接建立的时间损耗。

然后是**数据缓存层面**，对于`tools/list`这类不常变化的幂等性请求，每次都重新遍历插件列表是一种浪费。可以实现结果缓存机制，将查询结果缓存一段时间，热点请求能直接返回缓存数据，响应时间可降至亚毫秒级。

在**I/O层面**，当前的管道读写是阻塞操作，可以改为非阻塞模式，使用epoll或kqueue实现事件驱动，这样单线程就能支持数万级的并发连接。

**序列化层面**也有优化空间，JSON解析和序列化是CPU密集型操作。可以使用更快的JSON库如simdjson，或者对常用的响应进行预序列化，减少运行时的序列化开销。

最后是**横向扩展层面**，可以部署多个MCP Server实例，通过负载均衡器分发请求，实现水平扩展。同时需要确保服务设计是无状态的，方便进行弹性伸缩。

建议分阶段实施：第一阶段先上线线程池和结果缓存，快速提升吞吐量；第二阶段引入连接池和JSON优化，进一步降低延迟；第三阶段根据业务增长情况，考虑异步I/O和负载均衡架构。

#### 5）讲一下SSE通信模式，他和websocket有什么区别？

SSE（Server-Sent Events）是一种基于 HTTP/1.1 的服务器向客户端单向推送技术。客户端通过标准 HTTP 请求建立连接，服务器保持连接打开并持续发送事件数据。在 MCP 项目中，SSE 模式允许远程客户端通过网络接收服务器推送的实时通知和响应。它与 WebSocket 的核心区别在于通信方向——SSE 是单向的（服务器→客户端），而 WebSocket 是全双工双向通信；协议基础上，SSE 基于 HTTP，更容易穿透代理和防火墙，而 WebSocket 需要协议升级；重连机制方面，SSE 内置自动重连支持，而 WebSocket 需要手动实现；数据格式上，SSE 使用纯文本格式，WebSocket 支持二进制和文本。

在 MCP 项目中，SSE 主要用于远程工具调用和通知推送场景，而 STDIO 模式则用于本地进程间通信。选择 SSE 还是 WebSocket 取决于具体需求：如果只需要服务器主动推送数据且希望实现简单可靠，SSE 是更好的选择；如果需要双向实时交互，则应使用 WebSocket。

#### 6）怎么通过RAG对mcp工具的选择进行优化的？

项目通过 RAG（检索增强生成）技术对 MCP 工具选择进行了深度优化。核心思路是将传统的"全量工具发送给 LLM 判断"改为"**语义检索后只发送相关工具**"。

具体实现分为三个阶段：**首先**在服务启动时，将每个工具的名称、描述和参数信息组合成文本，调用阿里百炼的 Embedding API 转换为向量，**然后**存入向量索引中；当用户发起查询时，同样将查询文本向量化，通过余弦相似度搜索在索引中找到最相关的 Top-K 个工具；**最后**将这些工具转换为 LLM 可理解的函数调用格式发送给大模型。

为了提升效率，系统还加入了 **LRU 缓存机制**，避免重复调用 Embedding API，并设置了相似度阈值过滤不相关工具。这种优化显著降低了 Token 消耗，提高了工具选择的准确性，同时支持更多工具的扩展。

#### 7）工具调用时token消耗过高，是在什么时候呢？

Token 消耗过高主要发生在 Agent 需要了解和使用工具的场景中。当 Agent 刚启动或用户想了解有哪些可用工具时，系统会把所有工具的完整描述（包括名称、功能说明、参数 schema）发送给 LLM，这个阶段消耗的 Token 与工具数量成正比。当用户提出具体请求后判断该用哪个工具时，传统的做法是把用户 query 加上所有工具描述一并送给 LLM 让它从中挑选，如果工具数量很多但实际只需要其中几个，大量的 token 就浪费在了无关的工具信息上。工具执行完返回结果后，结果文本再传回 LLM 继续处理，这也会产生一定的 Token 消耗。

RAG-MCP 针对的就是工具选择这个阶段，通过语义检索先过滤出最相关的工具，只把这部分必要信息传给 LLM，显著降低了单次请求的 Token 消耗。

#### 8）RAG向量化的内容是什么？

RAG 向量化的是每个工具的结构化文本描述，包括工具名称、功能说明和参数信息。具体来说，向量化时会拼接出类似"Tool: calculator\nDescription: 执行数学计算\nParameters: expression (计算表达式)"这样的文本，然后调用 Embedding API 转换为向量存入索引。用户查询时，同样将查询文本向量化，通过余弦相似度与索引中的工具向量匹配，找到语义最相关的前 K 个工具返回。



#### 9）怎么把用户的问题，prompt去向量化的呢？有对比过引入RAG向量化后的效果吗？

用户问题向量化时，系统直接将用户输入的查询文本传给 Embedding 服务处理，流程很简单：先检查缓存是否有这个查询的结果，没有的话就调用阿里百炼 API 生成向量，然后存入缓存供后续使用。

关于效果对比，从架构设计上可以理论推算：假设有 50 个工具，优化前每次都要把所有工具描述发给 LLM，大概消耗上万 Token；引入 RAG 后只检索最相关的 3-5 个工具，Token 消耗可能降到几百，理论上能节省 90% 以上的 Token。不过从项目代码来看，目前没有找到实际的基准测试数据或 A/B 对比实验来验证这个效果。如果要证明优化效果，需要设计对照实验，分别统计引入 RAG 前后的 Token 消耗、工具选择准确率和接口响应延迟等指标。



#### 10）假设mcp服务端在Linux平台上运行时，CPU突然占满了，如何排查？

排查 CPU 占满的流程是先用 top 命令确认是哪个进程导致的，同时用 ps aux 找到具体的 PID。如果是多线程程序，可以用 top -Hp 查看线程级别信息。确认进程后，用 strace 追踪系统调用看是否有异常频繁的 syscall，比如看到大量 epoll_wait 说明在等待 I/O，如果是 clock_gettime 或 write/read 则可能是正常忙碌。还需要用 iostat 检查磁盘 I/O 和 netstat 查看网络统计，排除 I/O 阻塞导致的假象 CPU 高。

结合 MCP 场景，CPU 占满的常见原因包括：某个插件代码有死循环导致不断执行、请求 QPS 超过处理能力、nlohmann/json 序列化是 CPU 密集操作、RAG 向量化服务调用频繁导致 CPU 飙升、以及内存泄漏引发频繁的 new/delete。用 perf record 可以采样热点函数，定位到具体是哪个函数消耗最多 CPU。定位到问题后可以先 kill -9 重启服务止血，事后根据日志和 perf 报告做针对性优化，并考虑接入监控告警和 QPS 限制防止再次发生。



#### 11）你开发的MCP client是嵌入到Agent里面的吗？

从项目架构来看，MCP Client 是一个独立的客户端库，Agent 通过集成它来调用 MCP 工具，而不是直接嵌入到 Agent 内部。

具体来说，项目提供了 `MCPAgentIntegration` 类作为 MCP 与 Agent 的集成入口。Agent 初始化时创建这个集成器，设置 MCP Server 路径和相关配置，之后 Agent 发送请求时通过集成器调用 `MCPClient`，由客户端与 MCP Server 通信获取工具并执行。这种设计保持了模块解耦，MCP Client 可以独立升级，Agent 也方便替换不同的工具调用方案。



#### 12）你们是如何实现MCP协议的？协议包含哪些部分？

MCP 协议基于 JSON-RPC 2.0 标准实现，整体是一套轻量级的插件化架构。协议包含三个核心能力模块：Tools（工具调用）、Prompts（提示模板）和 Resources（资源访问），每个模块都有对应的 list 和 get/call 方法。同时包含 initialize、ping、shutdown 等生命周期方法用于连接管理和心跳检测。传输层支持 STDIO 和 SSE 两种模式，前者通过进程管道通信，适合本地调用，后者基于 HTTP SSE，适合远程场景。服务器还能通过通知机制主动向客户端推送消息。插件系统允许通过动态加载扩展协议能力，所有工具、提示和资源都通过统一路由分发。这种设计既保证了协议的简洁性，又提供了良好的扩展性。



#### 13）能介绍一下TCP/IP协议栈（七层或四层模型）吗？MCP协议对应到协议栈的哪一层？

TCP/IP 协议栈采用四层模型，从上到下分别是应用层、传输层、网络层和网络接口层。应用层负责为用户提供网络服务，常见协议有HTTP、FTP、SMTP等；传输层提供端到端的可靠数据传输，主要协议是TCP和UDP；网络层负责IP寻址和路由；网络接口层处理物理介质上的数据传输。

如果对照OSI七层模型，TCP/IP的四层对应应用层、传输层、网络层和数据链路层再加物理层。

MCP协议工作在应用层，基于HTTP传输（STDIO模式则通过进程管道），数据格式采用JSON-RPC 2.0。和gRPC、WebSocket这些协议一样，都位于应用层。区别在于MCP使用JSON文本格式，可读性好但性能不如gRPC的Protobuf二进制格式。MCP通过STDIO通信时甚至不经过网络层，只有SSE模式才会走到完整的TCP/IP协议栈。



#### 14）你做这个MCP项目的目的是什么，可以解决什么，不是有开源的可以用吗？

做这个 MCP 项目主要是出于几个实际考量。首先是有深度定制需求，开源实现面向通用场景，如果需要针对 AI Agent 做专门优化，比如 RAG 工具检索增强这类功能，通用方案不一定能覆盖。其次是通过完整实现一遍协议，能够深入掌握每个细节，便于后续问题排查和性能优化，也能根据业务变化快速调整。还有就是技术栈匹配的考虑，项目可能需要与现有的 C++ 技术栈深度集成，复用已有的日志系统、线程池、缓存模块等，引入外部开源库反而增加复杂度。另外自研也能避免开源项目维护变更带来的接口破坏风险，代码完全可控。

当然，开源方案确实存在，官方 MCP SDK 支持 TypeScript 和 Python 等语言，如果技术栈匹配直接用会更省事。但对于 C++ 项目，目前开源选择相对有限，这可能也是选择自研的现实原因之一。



#### 15）阐述MCP插件热加载的实现原理，并询问是否对比过其他热加载方案

MCP 插件热加载基于动态库加载实现，通过 `dlopen`（Linux）或 `LoadLibrary`（Windows）加载 .so/.dll 文件，然后获取插件导出的 `CreatePlugin` 和 `DestroyPlugin` 函数来创建和销毁实例。需要注意的是，当前实现是**启动时加载**，即服务启动时扫描插件目录并加载所有插件，并非运行时热加载。

关于是否对比过其他方案，从实现来看主要考量了跨平台兼容性，代码中针对 Linux、Windows、macOS 分别做了处理。如果需要实现真正的运行时热加载，可以考虑用 inotify 监控插件目录变化，或者提供 reload RPC 接口让外部触发重新加载。相比独立进程方案，动态库加载通信开销小且实现简单，但插件崩溃会影响主进程；而独立进程虽然隔离性好，但 IPC 通信会增加额外开销。



#### 16）了解RPC请求或系统交互中设置的具体超时时间，以及心跳检测的时长标准。

MCP 项目中的超时配置分为连接超时和请求超时两部分。连接超时设置为5秒，请求超时设置为30秒。STDIO 模式通过 condition_variable 的 wait_for 方法实现30秒的响应等待，而 SSE 模式则依赖 CURL 库设置的超时参数。

需要注意的是，项目当前没有实现显式的心跳检测机制，主要依赖 HTTP 层的 keep-alive 超时（5秒）和客户端的响应等待超时（30秒）来间接判断连接状态。业界常用的做法是心跳间隔10-30秒，超时阈值设为心跳间隔的3倍，如果在这个时间内没有收到心跳就判定为断连。如果业务对连接稳定性要求较高，可以考虑在服务器端增加定时 ping 消息，客户端在检测到超时后自动触发重连逻辑。



#### 17）提问RAG在实际业务中的文本分块（Chunking）策略是如何结合实际需求设计的。

当前项目的 RAG 向量化是以工具为粒度进行的，每个工具的名称、描述和参数信息组合成一段文本后单独向量化。这种方案简单直接，检索时返回的是完整工具，但在实际业务中可以进一步优化。

具体来说，可以将描述与参数分离后分别向量化，这样检索时能更精准地匹配用户意图。对于描述较长的工具，可以拆分成多个向量来保留更多细节。还可以在向量中加入工具类别、功能标签等元数据，便于在检索时进行过滤。比如 weather 工具可以分成"工具名称加类别描述"和"参数说明"两个块，用户问"北京明天天气"时，第一块负责匹配天气类工具，第二块负责匹配具体参数，这样检索准确率会更高。实际业务中常见的分块策略还包括固定长度分块、句子段落分块、语义分块和层次分块等，选择哪种策略需要根据文档结构、检索粒度和性能要求来权衡。
